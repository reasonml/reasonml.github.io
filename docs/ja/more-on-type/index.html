More on Type

型引数！

型はパラメータを受け取れます。他の言語で言うジェネリクスのようなものです。 それは型が引数を受け取り、新しい型を返す関数のようなものです！ パラメータは ' で始める必要があります。

パラメータ化された型のユースケースは重複をなくすことです。Before:

/* これは3要素のタプルで、次に説明があります。 */
type intCoordinates = (int, int, int);
type floatCoordinates = (float, float, float);

let buddy: intCoordinates = (10, 20, 20);

After:

type coordinates('a) = ('a, 'a, 'a);

/* 「型関数」の coordinates を適用すると、(int, int, int) が返ります。 */
type intCoordinatesAlias = coordinates(int);

let buddy: intCoordinatesAlias = (10, 20, 20);

/* または、より一般的に, インラインで書きます。 */
let buddy: coordinates(float) = (10.5, 20.5, 20.5);

実際には、型は推論されます。したがって、上記の例のより簡潔なバージョンはありません。

let buddy = (10, 20, 20);

型システムは上記を (int, int, int) に推論します。他には何も書く必要はありません。

型引数は至る所に表れます。

/* `list(string)` に推論されます */
let greetings = ["hello", "world", "how are you"];

型がパラメータを受け付けなかったとしたら（つまり、「型関数」がなかったとしたら）、標準ライブラリは listOfString、listOfInt、listOfTuplesOfInt、などの型を定義する必要があったでしょう。

型はより多くの引数を受け取ることが可能です。

type result('a, 'b) =
  | Ok('a)
  | Error('b);

type myPayload = {data: string};

type myPayloadResults('errorType) = list(result(myPayload, 'errorType));

let payloadResults: myPayloadResults(string) = [
  Ok({data: "hi"}),
  Ok({data: "bye"}),
  Error("Something wrong happened!")
];

相互再帰型

関数と同様、型も and を通して相互再帰することができます。

type student = {taughtBy: teacher}
and teacher = {students: list(student)};

注意 最初の行の最後にセミコロンがなく、2行目には type がありません。

設計方針

型引数のある型システムには、基本的に型レベルの関数があります。 list(int) は実際には int 型を受け取る list 型の関数です。そして、最終的に様々な場所で使用する具体的な型を返します。 あなたはこれが他の言語で多かれ少なかれ「ジェネリクス」と呼ばれているものだと気づくかもしれません。 例えば Java では ArrayList<Integer> です。

パワー最小の原則 は「物事を成し遂げる」時に適用します。 問題領域が許可するなら、絶対に最も抽象度の低い（最も具体的な）解決策を選択します。不安定で遠回りなことをドタバタやらずに、問題をより早く解決できます。 例えば、フリーフォームなデータよりも型を優先する、チューリング完全な関数呼び出しよりもデータ駆動の設定を優先する、マクロよりも関数呼び出しを優先する、プロジェクトの分岐よりもマクロを優先する、などです。 ドメインとパワーに制約があるときは、物事を分析しやすくなります。 つまり、ドメインがそれを許可するのに十分な制約がある場合です。

When a type system is an all-encompassing aspect of your program, we need to make sure we leave enough power in order not to overly constrain your expressiveness; without "type functions", you'd end up with quite a bit of boilerplate, e.g. hard-coded listOfInt, listOfString, listOfArrayOfFloat, their respective helper functions, etc. しかし、強力な型システムによってもたらされた力を過度に乱用しないようにしてください。 場合によっては、それ他の強力な型の必要性を減らすために、定型文を少し書くことは問題ありません。 それどころか、味わい深いトレードオフが、ファンシーな型よりも実用主義と判断を見せるかもしれません！
