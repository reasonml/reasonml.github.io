Convertir de JS

Preparación

Antes de proceder, ¡asegúrate de que Reason es lo que necesita tu equipo! A pesar de estar avalados por la popularidad de Reason y de BuckleScript, por favor no menosprecies innecesariamente a tus colegas y les des una mala primera impresión. Es difícil de deshacer después.

Esta guía cubre un flujo de trabajo que nos ha ayudado a nosotros en la conversión de forma rápida y eficaz. No se pretende repasar características FFI del lenguaje (aunque las pone en contexto). Se asumen conocimientos básicos de Reason/BuckleScript.

Sintaxis

Propósito: primero y destacado, hacer que el archivo sea sintácticamente válido. No se preocupe por tipos erróneos, módulos que faltan, incorrecta organización de los archivos, demasiados "external", etcetera. Regresaremos para corregir esto después de configurar la prueba de regresión que es "no más errores de sintaxis".

Como la sintaxis de Reason se asemeja bastante a la de JavaScript, en lugar de crear un archivo nuevo de Reason, simplemente copia un archivo js y trabaja con él.

Sugerencia: no olvides que puedes utilizar refmt en tu editor o terminal! Si no sabes, por ejemplo, la prioridad de algunas operaciones, escríbelas entre paréntesis como necesites, luego aplica refmt al código y mira cuantos quedan. De la misma manera, no necesitas perder el tiempo con indentaciones y espaciado; refmt se encarga de ellos.

/* copia del fichero JS original */
const school = require('school');

const defaultId = 10;

function queryResult(usePayload, payload) {
  if (usePayload) {
    return payload.student
  }
  return school.getStudentById(defaultId);
}

Estas son algunas de las cosas que harías en este punto:

  • Convertir la sintaxis de llamada a funciones.
  • Convertir var/const a let.
  • Ocultar los require.
  • Hacer estos cambios. Para expresiones idiomáticas que no tienen un equivalente en BuckleScript, usa bs.raw (documentación).

De nuevo, sólo preocúpate de hacer el archivo sintácticamente válido. Intentar aprender las tres sintaxis, tipos y otras semánticas mientras conviertes un archivo reduce un tercio tu velocidad de iteración.

/* sintácticamente válido, conversión semánticamente erronea */
/* const school = require('school'); */
let defaultId = 10;

let queryResult = (usePayload, payload) =>
  if (usePayload) {
    payload.student
  } else {
    /* no se necesita un retorno prematuro en Reason; if-else es una expresión */
    school.getStudentById(
      defaultId
    )
  };

Tipos, Paso 1

Objetivo: corregir los tipos, pero solo lo suficiente para ir al siguiente paso.

Es posible que todavía tengas errores de sintaxis ocasionales, pero no tan drásticos como en el paso anterior.

  • Cambiar foo.bar por foo##bar. Esta salida de escape característica de BuckleScript va a ser tu amigo a corto plazo.
  • Convertir {foo: bar} a [%bs.obj {foo: bar}] (docs). Después de refmt, esto se endulzará como {"foo": bar}.
  • Para comunicarse con archivos JS externos, utiliza external. Es un interface de función ajena de BuckleScript.
    • "externals" en código. No hay necesidad de crear diferentes archivos de código limpio para "externals" por ahora. Volveremos a hablar de esto.
    • Si es demasiado engorroso escribir correctamente los tipos para external de entrada y salida, usa tipos polimórficos substitutos, e.g. external getStudentById: 'whatever => 'whateverElse = ....
    • Para tipos de datos y patrones que son difíciles de convertir apropiadamente, ocasionalmente puedes crear convertidores como external unsafeCast : myPayloadType => anotherDataType = "%identity";.

Este es el primer paso; los tipos finales probablemente tendran un aspecto diferente. Por ahora, ¡a cosechar las recompensas! Una vez que corrijas todos los errores de tipos, tu archivo JS es generado. Mantenlo abierto uno al lado del otro. ¡Tiempo para volver y corregir todos los hacks!

/ * sintácticamente válida, todavía semánticamente erronea, pero mejor * 
[@bs.module "school"] external getStudentById : 'whatever => 'whateverElse = "getStudentById";

let defaultId = 10;

let queryResult = (usePayload, payload) =>
  if (usePayload) {
    payload##student /* this will be inferred as `Js.t 'a` */
  } else {
    getStudentById(defaultId)
  };

Semántica en tiempo de ejecución

Objetivo: corregir los errores de la salida JS generada.

Compáralo con tu antiguo archivo JS. La salida es probablemente incorrecta; Probablemente has convertido incorrectamente algunos expresiones idiomáticas y escrito mal algunas "externas".

  • Aplique tipos para dar forma a los objetos JS (las que requirieron ##).
  • Convierta cualquier parte a tipos registros o variantes de OCaml idiomático.

Todo el tiempo, comprueba la salida por si hay cambios.

tipo estudiante; /* tipo abstracto, descrito después */

[@bs.module "school"] external getStudentById : 'whatever => student = "getStudentById";

type payloadType = {. "student": student};

let defaultId = 10;

let queryResult = (usePayload, payload: payloadType) =>
  if (usePayload) {
    payload##student
  } else {
    getStudentById(defaultId)
  };

Limpieza (Tipos, paso 2)

Objetivo: haz tus tipos legítimos (conocido como, sound).

Vuelve atrás y corrige todo lo que hayas dejado durante el primer paso.

  • Asegúrate de que no has dejado ningún tipo 'whateverenexternal.
  • Puedes mantener los external en el código, o moverlos a un archivo.
/ * en el archivo actual */
type payloadType = {. "student": School.student}; /* TODO: ¡poner esto en otro sitio! */

let defaultId = 10;

let queryResult = (usePayload, payload: payloadType) =>
  if (usePayload) {
    payload##student
  } else {
    School.getStudentById(defaultId)
  };

/* en un fichero School.re dedicado */
type student;

[@bs.module "School"] external getStudentById : int => student = "getStudentById";
[@bs.module "School"] external getAllStudents : unit => array(student) = "getAllStudents";

Tipo student no tiene contenido real; eso se llama un tipo abstracto. Es una manera adecuada de especificar la relación entre llamadas externas sin saber como es la forma de los datos detrás de ella.

Y entonces ¡ya está!

Consejos

No trates de convertir totalmente un archivo JS en un archivo Reason impecable de un golpe. ¡Ese método en realidad podría ralentizarte! Está bien dejar "externals" y bs.obj, y temporalmente no aprovechar las atractivas características de OCaml (variantes, argumentos con etiquetas, etc.). Una vez has convertido unos pocos de otros archivos relacionados, puedes volver y ahora refactorizar más rápido confiando en el sistema de tipos.

Whatever nice utilities you find, put them in a tempUtils.re file or something. They're easy examples for your colleagues and removes some conversion churns.

Nosotros recomendamos encarecidamente que añadas el JS de salida a el control de versiones. Esto hace que la integración del sistemas de construcción sea cuasi-inexistente y se asegura de que cuando no estás allí, tus compañeros puedan hacer pequeños cambios, auditar la diferencia de la salida y detectar los fallos. También es un gran argumento de venta que la salida JS en el control de versiones sea amigable en caso de parches de emergencia en caliente (¡un gran punto de venta para directivos!). Incluso si estás actualizando la versión de BuckleScript, detectarías cualquier diferencia en la salida. Es como Jest snapshots, ¡gratis!
